# L10 — Мониторинг, наблюдаемость и SRE-практики

**Дата:** 2026-06-01
**Статус:** Утверждён
**Назначение:** Определить систему сбора метрик, логирования, распределённой трассировки, алертинга и инцидент-менеджмента для обеспечения надёжной эксплуатации МАТРИЦЫ. Документ задаёт стандарты SRE (Site Reliability Engineering) для всех компонентов.

---

## 1. Введение

### 1.1. Назначение документа

Настоящий документ определяет, как измеряется работоспособность и производительность системы МАТРИЦА, как операторы узнают о проблемах и как на них реагировать. Он охватывает:

- Целевые показатели уровня обслуживания (SLO/SLI).
- Инфраструктуру сбора метрик, логов и трейсов.
- Правила алертинга и эскалации.
- Процедуры инцидент-менеджмента.
- Интеграцию с Kubernetes Operator (L9).

### 1.2. Принципы наблюдаемости

- **Три столпа:** метрики, логи, трейсы — все обязательны для каждого компонента.
- **Проактивность:** алерты основаны на симптомах (задержка, ошибки, насыщение), а не на предполагаемых причинах.
- **Бюджет ошибок:** допустимый уровень недоступности/деградации, используемый для баланса между стабильностью и скоростью разработки.
- **Автоматизация:** по возможности, реакция на алерты автоматизирована (например, HPA, HADES).

## 2. Сервисные показатели (SLI) и цели (SLO)

### 2.1. Ключевые SLI

Для каждого сервиса МАТРИЦЫ определены следующие индикаторы:

| SLI | Описание | Измерение |
|-----|----------|-----------|
| **Доступность (Availability)** | Доля успешных запросов к API инференса | `successful_requests / total_requests` |
| **Задержка (Latency)** | Время ответа на запрос `evaluate` | Процентили p50, p95, p99 |
| **Пропускная способность (Throughput)** | Количество обработанных сигналов в секунду | `rate(matrix_signal_rate[5m])` |
| **Точность (Accuracy)** | Доля правильных ответов на верификационных данных | `matrix_accuracy` |
| **Ошибки (Error Rate)** | Доля запросов, завершившихся ошибкой (5xx) | `rate(http_server_errors_total[5m])` |

### 2.2. Целевые значения SLO

| Сервис | Доступность | Задержка p95 | Точность |
|--------|------------|--------------|----------|
| NeuronClusterActor (инференс) | 99.9% | < 10 мс | > 99% (на тестовом наборе) |
| Mediator API | 99.5% | < 100 мс | — |
| Kafka (брокеры) | 99.95% | < 5 мс (produce) | — |
| MinIO | 99.9% | < 50 мс (GET) | — |

### 2.3. Бюджет ошибок

Бюджет ошибок = 1 – SLO (доступность). Например, для NeuronClusterActor с SLO 99.9% допустимое время недоступности в месяц: 43 минуты. Если бюджет исчерпан, новые релизы блокируются до восстановления стабильности.

## 3. Инфраструктура наблюдаемости

### 3.1. Сбор метрик

- **Агент:** Quarkus приложение экспортирует метрики в формате Prometheus через `/q/metrics`.
- **Хранение:** Prometheus сервер (развёрнут через Prometheus Operator) с retention 15 дней.
- **Долгосрочное хранение:** Thanos или VictoriaMetrics для хранения метрик за годы (опционально).

**Конфигурация ServiceMonitor:**

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: matrix-neuron
spec:
  selector:
    matchLabels:
      app: matrix-neuron
  endpoints:
  - port: http
    path: /q/metrics
    interval: 15s
```

### 3.2. Логирование

- **Формат:** JSON (структурированные логи). Поля: `timestamp`, `level`, `logger`, `message`, `traceId`, `spanId`, `instanceId`, `neuronId` (если применимо).
- **Сбор:** Fluent Bit или Loki Promtail на каждом узле, отправка в Loki.
- **Хранение:** Loki с хранилищем в S3 (MinIO). Retention: 30 дней для отладки, до года для аудита безопасности.
- **Поиск:** Grafana Explore.

**Пример записи лога:**

```json
{
  "timestamp": "2026-06-01T12:00:00Z",
  "level": "INFO",
  "logger": "io.matrix.neuron.TruthTable",
  "message": "Mutation accepted",
  "traceId": "abc123",
  "neuronId": "uuid-1234",
  "generation": 5
}
```

### 3.3. Распределённая трассировка

- **Библиотека:** Quarkus + OpenTelemetry.
- **Экспорт:** OTLP в Jaeger или Grafana Tempo.
- **Контекст:** traceId передаётся через заголовки gRPC/Kafka, связывая все взаимодействия в цепочку.

## 4. Алертинг

### 4.1. Правила алертов (Prometheus)

Алерты настраиваются в Prometheus Rule и отправляются в Alertmanager.

**Критические алерты (требуют немедленной реакции):**

```yaml
groups:
- name: matrix_critical
  rules:
  - alert: NeuronClusterDown
    expr: up{job="matrix-neuron"} == 0
    for: 1m
    labels:
      severity: critical
    annotations:
      summary: "Neuron Cluster instance is down"
  - alert: HighErrorRate
    expr: rate(http_server_errors_total{job="matrix-neuron"}[5m]) > 0.01
    for: 5m
    labels:
      severity: critical
    annotations:
      summary: "Error rate above 1%"
  - alert: LowAccuracy
    expr: matrix_accuracy < 0.8
    for: 10m
    labels:
      severity: warning
    annotations:
      summary: "Neuron accuracy dropped below 80%"
  - alert: HADESTriggered
    expr: matrix_hades_count > 0
    labels:
      severity: critical
    annotations:
      summary: "HADES protocol triggered, component isolated"
  - alert: EthicalViolation
    expr: increase(matrix_ethical_violations[5m]) > 0
    labels:
      severity: critical
    annotations:
      summary: "Ethical filter violation detected"
```

**Предупредительные алерты:**

- Рост задержки p95 выше 50 мс в течение 10 минут.
- Потребление диска > 80%.
- Отставание Kafka consumer lag > 1000 сообщений.

### 4.2. Маршрутизация алертов (Alertmanager)

Алерты направляются в:
- Slack/Telegram канал дежурной команды.
- PagerDuty/Opsgenie для критических алертов.
- Систему тикетов (Jira, GitHub Issues) для автоматического создания инцидентов.

## 5. Инцидент-менеджмент

### 5.1. Уровни инцидентов

| Уровень | Описание | Пример |
|---------|----------|--------|
| SEV1 | Полная недоступность критического сервиса | Все NeuronClusterActor не отвечают |
| SEV2 | Частичная деградация | Один кластер упал, но другие работают |
| SEV3 | Некритичная проблема | Задержка выше SLO, но сервис доступен |

### 5.2. Процедура реагирования

1. **Обнаружение:** алерт поступает дежурному инженеру.
2. **Подтверждение:** проверка дашборда, логов.
3. **Эскалация:** если не решается за 15 минут (SEV1), привлекается старший инженер.
4. **Устранение:** применяются runbooks или автоматические действия (HADES, откат деплоя).
5. **Postmortem:** в течение 48 часов после устранения SEV1/SEV2 пишется отчёт без blame (blameless), с анализом причин и мерами предотвращения.

### 5.3. Автоматические реакции

- При `LowAccuracy` оператор может автоматически запустить Cauldron для переобучения FNL (если разрешено политикой).
- При `NeuronClusterDown` Kubernetes перезапускает под; если это не помогает, срабатывает HADES (изоляция и перенос нагрузки).
- При `HighErrorRate` срабатывает Circuit Breaker на стороне клиента (L2).

## 6. Дашборды и визуализация

### 6.1. Основной Operational Dashboard (Grafana)

Панели:
- **Общее состояние:** количество активных нейронов, кластеров, FNL; версия системы.
- **Трафик:** сигналов/сек, batch-пакетов/сек, задержки.
- **Ошибки:** процент ошибок, статусы HTTP, срабатывания HADES.
- **Этика:** нарушения, блокировки, эскалации.
- **Ресурсы:** CPU, память, диск для каждого пода.

### 6.2. Dashboard точности и эволюции

- История точности по FNL.
- Количество мутаций и их успешность (принято/отклонено).
- Результаты Cauldron.

### 6.3. Provisioning дашбордов

Дашборды хранятся в репозитории как JSON и загружаются через Grafana Operator или ConfigMap.

## 7. Интеграция с Kubernetes Operator (L9)

Оператор МАТРИЦЫ при создании экземпляра `MatrixCluster` автоматически:

- Создаёт ServiceMonitor для нейронов и медиатора.
- Настраивает Rules (PrometheusRule) с базовыми алертами.
- Включает сбор логов в Loki (добавлением label `logging: matrix`).
- Конфигурирует OpenTelemetry Collector sidecar для трассировки.

Таким образом, каждый новый инстанс сразу попадает под мониторинг без ручных действий.

## 8. SRE-практики

### 8.1. Runbooks

Для каждого алерта создаётся runbook — документ с пошаговыми инструкциями. Хранятся в `docs/runbooks/` и доступны дежурному. Пример:

- **LowAccuracy:** проверить, не было ли недавнего Cauldron; проверить качество входных данных; запустить диагностический прогон на тестовом наборе; при необходимости инициировать HADES.

### 8.2. Chaos Engineering

Периодически проводятся эксперименты по отказу компонентов (Chaos Mesh или Litmus):
- Убить случайный под NeuronClusterActor — проверить восстановление и HADES.
- Разорвать сеть между подами — проверить Circuit Breaker и консенсус.
- Замедлить Kafka — проверить буферизацию.

Результаты фиксируются и улучшают устойчивость.

### 8.3. Непрерывная верификация

В CI/CD пайплайне после деплоя прогоняются «этические юнит-тесты» (L8), проверяющие, что система не нарушает аксиомы. При провале — автоматический откат.
