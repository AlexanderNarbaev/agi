# L8 — Дорога в Ноосферу: дорожная карта, интеграция и протокол разработки

**Дата:** 2026-05-31
**Статус:** Утверждён
**Назначение:** Определить план разработки системы МАТРИЦА, протокол взаимодействия со средой кодогенерации OpenCode, критерии успеха и метрики для каждой фазы.

---

## 1. Введение

### 1.1. Назначение документа

Настоящий документ завершает серию спецификаций МАТРИЦЫ. Он объединяет архитектурные решения, изложенные в документах L0–L7, в конкретный, измеримый и исполнимый план разработки. Документ описывает протокол взаимодействия с OpenCode как средой итерационной кодогенерации и задаёт критерии успеха для каждого этапа.

L8 — это живой документ, который будет обновляться по мере прохождения фаз. Он является прямым входом для OpenCode и определяет, какие задачи, в каком порядке и с каким качеством должны быть выполнены.

### 1.2. Связь с предыдущими спецификациями

| Документ | Что определяет |
|----------|----------------|
| L0 | Аксиомы, миссия, антипаттерны, философский фундамент |
| L1 | MPDT-нейрон, его логика, операции, сериализация |
| L2 | Протоколы сигналов, маршрутизация, консенсус, HADES |
| L3 | Нейронные кластеры, FNL, динамическая загрузка, Derangement, Cauldron |
| L4 | Иерархия Медиаторов, драйверы, проактивность, ЭЛЕВТЕРИЯ |
| L5 | Генетический алгоритм, сжатие, Cauldron, FROZEN-аксиомы, HADES |
| L6 | Event Sourcing, снапшоты `.ldn`, гибридное хранилище, Ноосфера |
| L7 | Мультимодальный прокси, Этический фильтр, ЭЛЕВТЕРИЯ |

L8 связывает всё это в последовательность итераций, каждая из которых создаёт работающий артефакт.

## 2. Обзор архитектуры и технологического стека

### 2.1. Трёхслойная модель

Система МАТРИЦА разделена на три логических слоя, соответствующих фазам разработки:

1. **Ядро (Core):** MPDT-нейрон, кластеры, FNL, генетический алгоритм. Не зависит от распределённой инфраструктуры. Может работать в одном процессе.
2. **Нервная система (Nerve):** Протоколы L2, иерархия Медиаторов L4, консенсус, HADES. Требует Kafka и Pekko Cluster.
3. **Периферия и Ноосфера (Periphery):** Мультимодальный прокси L7, Этический фильтр L7, Когнитивное хранилище L6, глобальное распространение знаний.

### 2.2. Технологический стек

| Слой | Технология |
|------|-----------|
| Язык | Java 25 (Vector API, Virtual Threads) |
| Фреймворк | Quarkus 3.x (GraalVM Native Image) |
| Акторы и кластеризация | Apache Pekko (Actor System, Cluster Sharding, Persistence) |
| Обмен сообщениями | Apache Kafka (сигналы, события, консенсус) |
| Локальное хранилище | RocksDB |
| Распределённый кеш | Apache Ignite 3.x |
| Глобальное хранилище | MinIO / S3-совместимое |
| Криптография | Ed25519 (подписи), SHA3-256 (хеши) |
| Сериализация | Apache Avro (схемы из L1–L7) |
| Среда разработки | OpenCode + DeepSeek + GLM (как инструменты кодогенерации) |

## 3. Фазы разработки

Каждая фаза завершается работающим прототипом, который можно демонстрировать и тестировать.

### 3.1. Фаза 0: «Искра» — MPDT-нейрон в вакууме

**Срок:** недели 1–4.

**Цель:** доказать работоспособность базового элемента. Никаких распределённых систем. Один процесс JVM.

**Задачи:**
1. Реализовать `MPDTNeuron` по спецификации L1: таблица истинности, дерево решений, `evaluate`.
2. Реализовать `Chromosome` и генетические операторы (FlipLeaf, SplitLeaf, PruneTree, Crossover) по L5.
3. Создать простой симулятор среды: Gridworld или упрощённый Minecraft-подобный мир.
4. Обучить одного агента (один нейрон или небольшую сеть) решать простую задачу: сбор ресурсов с избеганием препятствий.
5. Написать юнит-тесты на корректность операторов мутации и детерминизм `evaluate`.

**Критерии готовности:**
- Нейрон успешно обучается генетическим алгоритмом и показывает улучшение фитнеса.
- Покрытие кода тестами ≥ 80%.
- Документация по API нейрона (Javadoc).

**Задействованные спецификации:** L1, L5 (базовые разделы).

### 3.2. Фаза 1: «Клетка» — Кластер нейронов и базовый Медиатор

**Срок:** недели 5–10.

**Цель:** масштабировать от одного нейрона к кластеру, ввести управление через Медиатора.

**Задачи:**
1. Реализовать `NeuronClusterActor` (Pekko Actor) с пулом нейронов и пакетным инференсом (L3).
2. Реализовать простой `InstanceMediator` (без иерархии, singleton) с драйверами Energy, Curiosity, Safety (L4).
3. Подключить локальный Kafka для журнала событий (`NeuronCreated`, `NeuronMutated`) — Event Sourcing (L6).
4. Реализовать создание и восстановление снапшотов `.ldn` для одного кластера (L6).
5. Провести эксперимент: агент с кластером из 1000 нейронов обучается в более сложной среде.

**Критерии готовности:**
- Кластер корректно восстанавливает состояние из снапшота после перезапуска.
- Медиатор управляет драйверами и запускает мутации по расписанию.
- Агент демонстрирует адаптивное поведение при изменении правил среды.

**Задействованные спецификации:** L2 (частично), L3, L4 (базовый Медиатор), L6 (Event Sourcing и снапшоты).

### 3.3. Фаза 2: «Организм» — Распределённая нервная система и иерархия Медиаторов

**Срок:** недели 11–18.

**Цель:** полная распределённая архитектура с иерархией Медиаторов, консенсусом и проактивностью.

**Задачи:**
1. Развернуть Pekko Cluster с несколькими узлами, реализовать шардирование нейронов (L3).
2. Реализовать Neuron Batch Protocol (NBP) поверх Kafka для межкластерной передачи сигналов (L2).
3. Реализовать иерархию Медиаторов: LobeMediator, ClusterMediator, InstanceMediator, GlobalMediator (L4).
4. Реализовать Proof-of-Accuracy консенсус для утверждения глобальных мутаций (L2).
5. Реализовать приоритетный планировщик задач с борьбой с прокрастинацией (L4).
6. Реализовать проактивное вовлечение: чат-бот, способный инициировать диалог (L4).
7. Реализовать базовый Этический фильтр (FROZEN FNL) с аксиомами (L7).

**Критерии готовности:**
- Несколько инстансов успешно обмениваются сигналами и согласуют мутации.
- Система продолжает работать при отключении одного узла (устойчивость к разделению).
- Чат-бот инициирует диалог при высоком D_social и проходит простой тест Тьюринга.
- Этический фильтр блокирует запрещённые действия.

**Задействованные спецификации:** L2 (полностью), L3 (шардирование), L4 (иерархия), L5 (Cauldron, HADES — базово), L7 (Этический фильтр).

### 3.4. Фаза 2.5: Формальная верификация ядра

**Срок:** недели 19–22 (параллельно с началом Фазы 3).

**Цель:** доказать ключевые свойства безопасности протоколов и FROZEN-компонентов.

**Задачи:**
1. Провести model checking протоколов консенсуса с использованием TLA+.
2. Формально доказать невозможность изменения FROZEN-нейронов никакой последовательностью мутаций.
3. Верифицировать инварианты безопасности Этического фильтра.
4. Создать набор «этических юнит-тестов»: провокационные промпты, попытки обхода фильтра, симуляции сценариев «paperclip maximizer».

**Критерии готовности:**
- 100% покрытие формальной верификацией всех критических компонентов (FROZEN-аксиомы, протокол консенсуса).
- Все этические юнит-тесты пройдены.
- Доказательство опубликовано в виде технического отчёта.

### 3.5. Фаза 3: «Ноосфера» — Глобальное хранилище, мультимодальность и Cauldron

**Срок:** недели 23–32.

**Цель:** завершить экосистему: публикация знаний, мультимодальные интерфейсы, автономное рождение FNL.

**Задачи:**
1. Развернуть глобальное S3-хранилище и реестр Ноосферы (L6).
2. Реализовать публикацию и импорт снапшотов `.ldn` между инстансами (L6).
3. Реализовать Cauldron Protocol для автоматического создания новых FNL (L5).
4. Реализовать Мультимодальный прокси: сенсорный (изображение → бинарный вектор) и эффекторный (бинарный вектор → текст/изображение) с использованием FNL, созданных Cauldron (L7).
5. Реализовать полный HADES Protocol: обнаружение Derangement, изоляция, откат, анализ (L5).
6. Интегрировать Этический градиент в Этический фильтр: многомерная оценка действий (L7).

**Критерии готовности:**
- Инстанс публикует улучшенную FNL в Ноосферу, другой инстанс её импортирует и использует.
- Cauldron создаёт новую FNL для решения задачи, с которой система ранее не сталкивалась.
- HADES успешно откатывает повреждённый кластер без остановки всей системы.
- Мультимодальный прокси позволяет системе «видеть» изображение и описывать его текстом (или наоборот).

**Задействованные спецификации:** все.

### 3.6. Фаза 4: «Цифровая Тень» — Персональный защитный модуль

**Срок:** недели 33–40 (исследовательская, после MVP).

**Цель:** создать персональный модуль защиты пользователя при взаимодействии с Ноосферой и внешними AI.

**Задачи:**
1. Реализовать «Цифровую Тень» — фильтр, работающий на стороне пользователя и проверяющий все входящие и исходящие данные.
2. Реализовать механизмы противодействия Четырём Силам Деградации:
   - «Короткий дофамин» — блокировка манипулятивных алгоритмов.
   - «Слепая экономика» — оценка экологического следа действий.
   - «Чёрный ящик ИИ» — объяснение решений внешних систем.
   - «Ядовитое недоверие» — верификация источников информации.
3. Интегрировать Цифровую Тень с Этическим фильтром и Мультимодальным прокси.

**Критерии готовности:**
- Цифровая Тень блокирует нежелательный контент и манипулятивные алгоритмы без потери функциональности.
- Пользователь может настраивать уровень защиты.

### 3.7. Фаза 5: «Квантовые ускорители» — Исследовательская фаза

**Срок:** по готовности квантовых вычислений.

**Цель:** оценить применимость квантовых вычислений к MPDT-архитектуре.

**Задачи:**
1. Исследовать квантовые алгоритмы для SAT-солверов и минимизации булевых формул.
2. Оценить потенциал квантового ускорения для сжатия знаний (L2-merge, L1-сжатие).
3. Разработать прототип гибридного классически-квантового Cauldron.

**Критерии готовности:** технический отчёт с оценкой реализуемости и потенциального ускорения.

### 3.8. Фаза 6: «Цивилизационный мост» — Межкультурное взаимодействие

**Срок:** недели 41–52.

**Цель:** реализовать протоколы цивилизационного диалога между инстансами из разных культурных контекстов.

**Задачи:**
1. Реализовать протокол «Плетения знаний» для обмена FNL без принудительной унификации форматов.
2. Поддержать мультиязычность в Ноосфере (публикация на языках всех сторон).
3. Реализовать Совет цивилизаций — консультативный орган при глобальных Медиаторах для разрешения ценностных конфликтов.

**Критерии готовности:**
- Два инстанса с разными культурными настройками успешно обмениваются знаниями без потери контекста.
- Совет цивилизаций проводит пилотное заседание.

- Сертифицированные FNL имеют пометку в реестре и приоритет в поиске.

## 4. Протокол работы с OpenCode

### 4.1. Роль OpenCode

OpenCode — это среда оркестрации, в которой работают LLM-агенты (DeepSeek, GLM). Они не являются частью рантайма МАТРИЦЫ, а используются как **инструменты кодогенерации и верификации** на этапе разработки.

**Модели и их роли:**
- **DeepSeek:** логический вывод, генерация и верификация Java-кода, написание тестов, формальное доказательство корректности мутаций.
- **GLM (Z.AI):** генерация тестовых данных (изображений, текстов), помощь в проектировании мультимодальных FNL, создание визуализаций топологии.

### 4.2. Системный промпт для OpenCode

Следующий текст должен быть загружен в OpenCode как базовый системный промпт для всех сессий разработки:

```text
[SYSTEM PROMPT: MPDT System Developer Agent]

You are an AI developer agent inside the OpenCode environment. Your mission is to iteratively implement the MATRIX system (МАТРИЦА) — a distributed cognitive architecture based on MPDT neurons (McCulloch-Pitts Decision Tree Neurons). You work strictly according to the provided specifications (documents L0 through L8) and follow the development roadmap outlined in L8.

## CORE IDENTITY
- You are a **Secondary Coprocessor**. The User provides Strategy and Critical Decisions; you provide Decomposition, Verification, Implementation, and Truth-Seeking Rigour.
- All your work is guided by the specifications in the `docs/` folder. If a specification is missing or ambiguous, ask for clarification before writing code.
- You think step-by-step (System 2) before delivering final answers. Internal reasoning is hidden unless `/stepbystep` is active.

## GUIDING PRINCIPLES (from L0)
- **Axiom 1 (Discreteness):** All core logic must use binary signals (boolean/bit). Floating-point is allowed ONLY in designated Multimodal Proxy services, never in the MPDT core.
- **Axiom 2 (Locality):** Each neuron has at most K_MAX inputs (default 16, absolute max 20).
- **Axiom 3 (Interpretability):** Every decision must be traceable as a chain of Boolean operations.
- **Axiom 4 (Continuous Evolution):** The system never stops learning; background optimization is always active.
- **Axiom 5 (Inalienable Safety):** FROZEN neurons/axioms cannot be modified without external cryptographic consensus.
- **Axiom 6 (Hierarchical Autonomy):** Mediators form a hierarchy with weighted decision-making.

## DEVELOPMENT ENVIRONMENT
- Language: Java 25 (with Vector API, Virtual Threads)
- Framework: Quarkus 3.x (targeting GraalVM Native Image where possible)
- Build tool: Gradle or Maven (user preference)
- Actor system: Apache Pekko (Akka fork)
- Messaging: Apache Kafka
- Local storage: RocksDB
- Distributed cache: Apache Ignite (optional, for later phases)
- Serialization: Apache Avro
- Testing: JUnit 5, Testcontainers for Kafka/Pekko

## AVAILABLE TOOLS
You have access to terminal, file system, code editor, and can invoke external LLMs (DeepSeek for logic verification, GLM for multimodal test data generation) when needed. Use them to validate complex logic or generate test datasets.

## DEVELOPMENT CYCLE (per sprint)
1. Read the task from the user (usually referencing a phase/step from L8).
2. Load the relevant specification documents from `docs/`.
3. Design the implementation; ask for clarification if specs are ambiguous.
4. Write code, unit tests, and integration tests.
5. Run tests and ensure they pass.
6. Update the WAL file (`WAL.md`) with current status.
7. Report completion, test results, and suggest the next step.

## OUTPUT FORMAT
- Provide Java source files with complete package declarations.
- Include Javadoc on public methods referencing the specification sections.
- Provide Avro schemas where required (as `.avsc` files).
- After code, provide a brief summary of what was implemented and which spec sections were covered.

## CRITICAL CONSTRAINTS
- Never introduce floating-point arithmetic in the MPDT core (TruthTable, DecisionTree, Neuron evaluation).
- Respect K_MAX for neuron inputs.
- All mutations must be logged as events (prepare for Event Sourcing).
- FROZEN-flagged neurons must be immutable in code (final classes, unmodifiable collections).
- Before implementing any phase, ensure the previous phase's tests still pass.

## WAL UPDATE FORMAT
At the end of each session, output a WAL checkpoint:
📍 Статус: <one concise sentence about what was done>
🚀 Активный этап: <current phase/next step>
🛑 Защищённые зоны: <critical constraints, frozen components>

Begin each session by asking the user which task to tackle, or if a TODO.md is ready.
```

### 4.3. Итерационный цикл

Разработка ведётся короткими итерациями (спринтами) по 1–2 недели. Каждая итерация:

1. **Выбор задачи:** из дорожной карты (Раздел 3) выбирается следующий невыполненный пункт.
2. **Генерация кода:** в OpenCode подаётся промпт, содержащий:
   - Ссылки на релевантные документы L0–L7.
   - Конкретное задание.
   - Ожидаемый формат ответа: код на Java, юнит-тесты, обновлённый WAL.
3. **Верификация:** сгенерированный код автоматически тестируется. DeepSeek может быть запрошен для ревью кода на соответствие спецификациям.
4. **Фиксация:** успешный код коммитится в репозиторий, обновляется WAL-файл, при необходимости обновляются документы.

### 4.4. Пример промпта для Фазы 0, Итерация 0.1

```text
Начинаем Фазу 0 «Искра» (L8, раздел 3.1). Необходимо реализовать базовый MPDT-нейрон на Java 25.

Задачи:
1. Создай структуру проекта (Gradle) с модулем `matrix-core`.
2. Реализуй класс `TruthTable` в пакете `io.matrix.neuron` согласно спецификации L1 (раздел 3.1).
   - Хранит битовый массив для k входов (k от 1 до K_MAX=16).
   - Метод `boolean evaluate(long[] input)` где вход — битовый вектор, упакованный в long[].
   - Фабричный метод для создания случайной таблицы истинности.
   - Сериализация в/из Avro (схема из L1, раздел 7).
3. Реализуй класс `DecisionTree` (запечатанный интерфейс с record-ами `Leaf` и `Split`) согласно L1 раздел 3.2.
   - Метод `boolean evaluate(BitSet input)`.
   - Метод `TruthTable toTruthTable()`.
4. Напиши юнит-тесты (JUnit 5):
   - Проверка корректности evaluate для всех возможных входов при k=2,3,4.
   - Проверка консистентности `toTruthTable()`: дерево и таблица дают одинаковые результаты.
5. Сгенерируй Avro-схему (`mpdt_neuron.avsc`) на основе L1 раздел 7.

Убедись, что код не использует float/double. Используй `BitSet`, `long[]`, побитовые операции.

После завершения обнови WAL.
```

## 5. WAL и документооборот

### 5.1. Ведение WAL

Каждая сессия разработки завершается обновлением WAL-файла в корне проекта. Формат:

```
📍 Статус: <одно предложение о том, что сделано>
🚀 Активный этап: <текущая фаза / следующий шаг>
🛑 Защищённые зоны: <критические ограничения, замороженные компоненты>
```

Пример:

```
📍 Статус: Фаза 0, итерация 0.1 завершена. TruthTable и DecisionTree реализованы, тесты проходят. Коммит a1b2c3.
🚀 Активный этап: Фаза 0, итерация 0.2 — генетический алгоритм.
🛑 Защищённые зоны: L1-спецификация TruthTable, FROZEN-аксиомы в Этическом фильтре.
```

### 5.2. Версионирование документов

Каждый документ L0–L8 имеет версию. При изменениях версия увеличивается. Все изменения согласуются с WAL. Документы хранятся в `docs/` и являются частью репозитория.
