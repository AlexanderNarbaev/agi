# L2 — Протоколы взаимодействия, маршрутизации и консенсуса

**Дата:** 2026-05-31
**Статус:** Утверждён
**Назначение:** Определить коммуникационную инфраструктуру системы МАТРИЦА: протоколы передачи сигналов, адресацию, консенсус и криптографическую целостность.

---

## 1. Введение

### 1.1. Назначение документа

Настоящий документ определяет, как нейроны, нейронные кластеры и медиаторы системы МАТРИЦА обмениваются сигналами, как сигналы маршрутизируются в масштабируемой сети, и как распределённые узлы достигают согласия относительно эволюции системы, не жертвуя безопасностью и живучестью.

### 1.2. Ключевые принципы

- **Асинхронность:** все взаимодействия асинхронны и реализованы через обмен сообщениями. Блокирующие вызовы отсутствуют.
- **Пакетная передача:** сигналы агрегируются для минимизации сетевого трафика.
- **Иерархическая маршрутизация:** глобальная DHT-сеть связывает кластеры; локальная диспетчеризация внутри кластера не требует сетевого трафика.
- **Консенсус на основе Proof-of-Accuracy:** решения принимаются голосованием, где вес узла пропорционален его реальной полезности.
- **Криптографическая целостность:** каждое сообщение подписывается; ключи регулярно обновляются через консенсус.
- **Устойчивость к разделению:** система продолжает работу в условиях сетевых сбоев, разрешая конфликты при восстановлении связи.

## 2. Модель коммуникации: сигналы

### 2.1. Определение сигнала

Сигнал — это единица информации, передаваемая от одного нейрона другому. Значение всегда бинарно: `0` или `1`.

**Формат сигнала (логический уровень):**

```
Signal {
    senderId:   NeuronId       // кто отправил
    receiverId: NeuronId       // кому адресован
    value:      bit            // 0 или 1
    timestamp:  uint64         // время возникновения (по локальным часам отправителя)
    priority:   uint8          // 0 (нормальный) или 1 (высокий, для критических сигналов)
}
```

### 2.2. Жизненный цикл сигнала

1. **Генерация:** MPDT-нейрон в результате инференса выставляет значение на выходном порте.
2. **Локальная агрегация:** все исходящие сигналы нейронов одного кластера собираются локальным диспетчером в пакет.
3. **Глобальная маршрутизация:** пакет отправляется в кластер-получатель через систему обмена сообщениями Apache Kafka.
4. **Локальная доставка:** диспетчер кластера-получателя распределяет сигналы конкретным нейронам.
5. **Обработка:** нейрон-получатель учитывает сигнал при следующем цикле инференса.

## 3. Протокол пакетной передачи сигналов (Neuron Batch Protocol)

Для масштаба в триллионы нейронов индивидуальная отправка сигналов невозможна. NBP решает эту проблему.

### 3.1. Агрегация в кластере-отправителе

Каждый `NeuronClusterActor` периодически (с интервалом `T_batch`, по умолчанию 5 мс) собирает все сгенерированные сигналы за этот период, группирует их по целевому кластеру и формирует пакеты.

**Структура пакета:**

```
NeuronBatch {
    batchId:          UUID
    sourceClusterId:  ClusterId
    targetClusterId:  ClusterId
    signals:          List<Signal>
    compression:      CompressionMethod
    signature:        bytes         // Ed25519 подпись отправителя
}
```

### 3.2. Сжатие пакетов

Поскольку значения сигналов бинарны, применяются специальные методы:

- **Run-Length Encoding (RLE):** кодирует последовательности одинаковых значений.
- **Битовые маски:** для каждого нейрона-получателя хранится бит наличия сигнала, затем значения только для активных.
- **LZ4:** для часто повторяющихся паттернов нейронных ID.

Выбор метода определяется адаптивно в зависимости от свойств трафика.

### 3.3. Транспорт

Пакеты передаются через Apache Kafka. Топик для каждого кластера-получателя: `neuron-batch.{targetClusterId}`. Это обеспечивает надёжную доставку, сохранение порядка и масштабирование через партиции.

## 4. Адресация и маршрутизация

### 4.1. Идентификаторы

Каждый `NeuronCluster` имеет уникальный `ClusterId` (UUID). Нейроны идентифицируются `NeuronId` (UUID + поколение). Кластер, которому принадлежит нейрон, определяется при создании; информация хранится в глобальном реестре.

### 4.2. Глобальная DHT-маршрутизация

Для нахождения целевого кластера по `NeuronId` используется распределённая хеш-таблица (DHT) на основе протокола Kademlia, реализованная поверх Pekko Cluster.

**Принцип:**
- Пространство ключей: 160-битный хеш (SHA-1) от `NeuronId`.
- Узлы DHT: все `NeuronClusterActor`-ы.
- Каждый узел хранит таблицу маршрутизации и отвечает за диапазон ключей.

### 4.3. Гибридная маршрутизация

Для критичных по времени сигналов используются предварительно установленные каналы (выделенные топики Kafka) с кэшированием маршрутов. DHT-поиск применяется только для новых или редко используемых связей. Локальные связи внутри FNL не требуют сетевого трафика.

**Целевые показатели задержек:**
- Локальный инференс: <10 нс на нейрон.
- Внутрикластерный сигнал: <1 мкс.
- Межкластерный (Kafka): <10 мс.
- DHT-lookup (без кэша): <100 мс.

### 4.4. Локальная диспетчеризация

Внутри `NeuronClusterActor`-а нейроны хранятся в массиве или хеш-таблице. При получении пакета кластер распаковывает сигналы и доставляет их напрямую нужным нейронам без сериализации.

## 5. Топология связей

### 5.1. Локальные связи (внутри кластера)

Нейроны внутри одного кластера находятся в общей памяти. Связи между ними реализуются напрямую, без сетевого трафика и криптографической подписи (доверие внутри кластера).

### 5.2. Глобальные связи (между кластерами)

Глобальные связи эмулируются через протокол NBP. Нейрон может иметь входы от нейронов других кластеров. Для нейрона-получателя источник сигнала выглядит как виртуальный вход, неотличимый от локального (кроме задержки).

**Ограничение:** нейрон может иметь не более `G_MAX` глобальных входов (по умолчанию 4), остальные должны быть локальными. Это гарантирует, что большинство связей остаются быстрыми.

## 6. Протокол консенсуса: Proof-of-Accuracy

### 6.1. Зачем нужен консенсус

Распределённые узлы должны договариваться о:
- Принятии или отклонении мутации.
- Создании или удалении нейронных кластеров.
- Обновлении протоколов.
- Замене криптографических ключей.

### 6.2. Proof-of-Accuracy (PoA)

Вес голоса узла при консенсусе пропорционален его точности на проверочных данных за последний период:

```
weight_i = f(accuracy_i, uptime_i, contribution_i)
```

где:
- `accuracy_i` — скользящее среднее точности предсказаний узла на отложенной выборке,
- `uptime_i` — время безотказной работы (нормированное),
- `contribution_i` — число успешных мутаций, внедрённых узлом в общую сеть.

Функция `f` детерминирована и верифицируема всеми участниками.

### 6.3. Процесс голосования

1. **Предложение:** любой узел отправляет предложение (`Proposal`) в топик `consensus.proposals`.
2. **Сбор голосов:** в течение окна `T_consensus` узлы вычисляют свой вес и отправляют подписанный голос (`Vote`).
3. **Подсчёт:** по истечении окна каждый узел подсчитывает голоса. Если суммарный вес "за" превышает порог (2/3 от общего веса проголосовавших), предложение принято.
4. **Применение:** изменение записывается в распределённый реестр и рассылается всем узлам.

### 6.4. Иерархический консенсус

Решения классифицируются по уровню воздействия:

| Уровень | Зона воздействия | Кто принимает решение |
|---------|------------------|----------------------|
| Level 0 | Одна FNL | LobeMediator единолично |
| Level 1 | Один кластер | ClusterMediator с уведомлением InstanceMediator |
| Level 2 | Один инстанс | InstanceMediator, с возможностью оспаривания пользователем |
| Level 3 | Все инстансы | Совет глобальных Медиаторов, порог 2/3 от суммарного веса |

### 6.5. Разрешение форков

При сетевом разделении могут возникнуть конкурирующие мутации. При восстановлении связи узлы сравнивают цепочки поколений и выбирают ту, которая имеет больший суммарный вес голосов (подтверждённую PoA).

## 7. Криптографическая подпись и реестр

### 7.1. Подпись сообщений

Каждое сообщение (сигнал, пакет, голос) подписывается закрытым ключом отправителя. Используется алгоритм Ed25519 для высокой производительности и короткой подписи.

**Процесс:**
- При создании кластер генерирует пару ключей.
- Открытый ключ регистрируется в Neuron Identity Ledger.
- При получении сообщения получатель проверяет подпись, используя открытый ключ из реестра.

### 7.2. Эволюция ключей

Для предотвращения компрометации долгоживущих ключей:
- Каждые `T_key_rotation` (по умолчанию 7 дней) кластер генерирует новую пару ключей.
- Новый открытый ключ подписывается старым закрытым ключом и предлагается в консенсус.
- После одобрения новый ключ записывается в реестр, старый отзывается.

### 7.3. Neuron Identity Ledger

Распределённый реестр, хранящий:
- Открытые ключи всех легитимных кластеров.
- Историю смены ключей.
- Хеши одобренных мутаций и снапшотов.

Реализован как компактифицированный топик Kafka, где ключом является идентификатор кластера, а значением — текущий открытый ключ и метаданные.

## 8. HADES Protocol: контролируемый перезапуск

### 8.1. Триггеры HADES

- Точность кластера или FNL падает ниже критического порога `ACC_CRITICAL` (по умолчанию 0.5) на протяжении `T_critical` циклов.
- Обнаружено неразрешимое противоречие в журнале событий.
- Зафиксировано состояние "Derangement" (неконтролируемое отклонение поведения), которое не удаётся исправить локально.

### 8.2. Процедура

1. **Изоляция:** проблемный компонент изолируется. Сигналы к нему и от него блокируются.
2. **Диагностика:** вышестоящий Медиатор анализирует журналы и метрики.
3. **Сохранение:** текущее состояние и все несохранённые события записываются в аварийное хранилище.
4. **Откат:** компонент откатывается до последнего стабильного снапшота.
5. **Восстановление:** функциональность возвращается поэтапно, с верификацией на каждом шаге.
6. **Анализ:** Медиатор записывает причину сбоя в глобальный журнал опыта.

## 9. Устойчивость к разделению (CAP/PACELC)

### 9.1. Выбор модели

МАТРИЦА выбирает **AP (доступность + устойчивость к разделению)** с eventual consistency. Сетевые разделения в глобальной системе неизбежны.

### 9.2. Работа при разделении

- Каждый узел продолжает обрабатывать сигналы и обучаться локально.
- Мутации, созданные во время разделения, получают временный статус "неподтверждённых".
- После восстановления связи запускается процедура примирения:
  - Узлы обмениваются журналами событий.
  - Конфликтующие изменения разрешаются через PoA-голосование.
  - Победившие мутации вливаются в основную ветвь.

### 9.3. PACELC

В отсутствие разделения система выбирает **Latency (быстродействие)**, а не Consistency. Операции чтения и записи не ждут подтверждения от всех реплик, а выполняются локально с асинхронной репликацией. Это обеспечивает минимальную задержку инференса.

## 10. Обработка ошибок и Circuit Breaker

Каждый узел отслеживает метрики взаимодействия с другими кластерами: доля успешных запросов, средняя задержка. Если за заданный интервал доля ошибок превышает порог `ERROR_THRESHOLD` (по умолчанию 50%), цепь размыкается:

- Попытки отправки в проблемный кластер прекращаются на время `COOLDOWN_PERIOD`.
- Сигналы буферизируются или перенаправляются на резервный кластер.
- После истечения периода охлаждения выполняется пробный пакет. При успехе цепь замыкается; иначе — снова размыкается.

## 11. Форматы сообщений (Avro-схемы)

### 11.1. Сигнал

```avro
{
  "type": "record",
  "name": "Signal",
  "namespace": "io.matrix.protocol",
  "fields": [
    {"name": "senderId", "type": {"type": "record", "name": "NeuronId",
      "fields": [
        {"name": "uuid", "type": "string"},
        {"name": "generation", "type": "long"}
      ]
    }},
    {"name": "receiverId", "type": "NeuronId"},
    {"name": "value", "type": "boolean"},
    {"name": "timestamp", "type": "long"},
    {"name": "priority", "type": "int"}
  ]
}
```

### 11.2. Пакет (NeuronBatch)

```avro
{
  "type": "record",
  "name": "NeuronBatch",
  "namespace": "io.matrix.protocol",
  "fields": [
    {"name": "batchId", "type": "string"},
    {"name": "sourceClusterId", "type": "string"},
    {"name": "targetClusterId", "type": "string"},
    {"name": "signals", "type": {"type": "array", "items": "Signal"}},
    {"name": "compression", "type": {"type": "enum", "name": "Compression",
      "symbols": ["NONE", "LZ4", "RLE", "BITMASK"]
    }},
    {"name": "signature", "type": "bytes"}
  ]
}
```

### 11.3. Предложение консенсуса

```avro
{
  "type": "record",
  "name": "Proposal",
  "namespace": "io.matrix.consensus",
  "fields": [
    {"name": "proposalId", "type": "string"},
    {"name": "type", "type": {"type": "enum", "name": "ProposalType",
      "symbols": ["MUTATION", "TOPOLOGY_CHANGE", "KEY_ROTATION", "PROTOCOL_UPGRADE"]
    }},
    {"name": "payload", "type": "bytes"},
    {"name": "proposerId", "type": "string"},
    {"name": "timestamp", "type": "long"},
    {"name": "signature", "type": "bytes"}
  ]
}
```

## 12. Заключение

Документ L2 задаёт полную спецификацию нервной системы МАТРИЦЫ: от мельчайшего сигнала до глобального консенсуса. Асинхронная событийная архитектура в сочетании с иерархической маршрутизацией и пакетной передачей позволяет масштабироваться до триллионов нейронов без экспоненциального роста трафика. Протокол Proof-of-Accuracy гарантирует, что эволюция системы направляется реальной пользой, а криптографическая подпись и регулярная смена ключей обеспечивают доверие в ненадёжной среде. HADES Protocol и Circuit Breaker завершают картину, делая систему устойчивой к отказам и атакам.
