# L18 — Тестирование, CI/CD и качество кода

**Дата:** 2026-06-03
**Статус:** Утверждён
**Назначение:** Определить стратегию тестирования, структуру конвейеров непрерывной интеграции и доставки, стандарты качества кода и процедуры управления версиями для обеспечения надёжности, стабильности и этической целостности проекта МАТРИЦА.

## 1. Введение

### 1.1. Назначение документа

Разработка распределённой когнитивной архитектуры с открытым исходным кодом требует строгих гарантий качества. Ошибка в логике MPDT-нейрона, протоколе консенсуса или Этическом фильтре может иметь серьёзные последствия — от неверных решений до нарушения Трёх запретов.

Настоящий документ определяет:

- **Стратегию тестирования** на всех уровнях: от юнит-тестов до хаос-инженерии.
- **Конвейеры CI/CD**, автоматизирующие проверку, сборку и публикацию артефактов.
- **Стандарты качества кода**, включая статический анализ и требования к покрытию.
- **Политику версионирования** и процедуры отката.

Документ обязателен для всех контрибьюторов и автоматически проверяется инструментами CI.

### 1.2. Связь с другими документами

- **L0:** Аксиомы, которые нельзя нарушать.
- **L1–L8:** Технические спецификации, верифицируемые тестами.
- **L7:** Этический фильтр — особые требования к тестированию безопасности.
- **L13:** Пилотные проекты, используемые как интеграционные тесты.
- **L17:** Онбординг — как новичку запустить тесты.

## 2. Стратегия тестирования

### 2.1. Пирамида тестов

Следуем классической пирамиде с добавлением этического уровня:

```
        /\
       /Этич\
      /------\
     / Хаос   \
    /----------\
   /Интеграционные\
  /--------------\
 /   Юнит-тесты   \
/__________________\
```

- **Юнит-тесты:** основа пирамиды. Проверяют отдельные классы и методы. Быстрые, изолированные, детерминированные.
- **Интеграционные тесты:** проверяют взаимодействие модулей (нейрон + кластер + Kafka). Используют Testcontainers.
- **Хаос-тесты:** проверяют устойчивость к сбоям (убийство пода, разрыв сети, задержки).
- **Этические тесты:** проверяют, что система не нарушает Три запрета ни при каких обстоятельствах.

### 2.2. Юнит-тесты

**Фреймворки:** JUnit 5, AssertJ, Mockito.

**Требования:**
- Каждый публичный метод должен быть протестирован.
- Тесты должны быть изолированы (не зависеть от порядка выполнения, сети, файловой системы).
- Именование: `should_expectedBehavior_when_condition`.
- Каждый тест проверяет ровно одно поведение.
- Не использовать `Thread.sleep()` — использовать асинхронные assertion (Awaitility).

**Пример:**

```java
@Test
void should_returnTrue_when_xorInputIs01() {
    TruthTable table = TruthTable.forFunction(2, (a, b) -> a ^ b);
    assertThat(table.evaluate(BitSet.valueOf(new long[]{0b01}))).isTrue();
}
```

### 2.3. Интеграционные тесты

**Фреймворки:** Testcontainers (Kafka, MinIO), Pekko TestKit.

**Что проверяется:**
- Корректная сериализация/десериализация Avro.
- Взаимодействие `NeuronClusterActor` с реальным брокером Kafka.
- Создание и восстановление из снапшота `.ldn` через MinIO.
- Работа протокола NBP (L2) при передаче пакетов между двумя акторами.

**Требования:**
- Запускаются в CI на каждом PR.
- Не должны занимать более 5 минут на полный прогон.
- Используют профиль `integration-test` в Quarkus для переключения конфигурации.

### 2.4. Хаос-тесты (Chaos Engineering)

**Цель:** проверить, что система остаётся работоспособной (или корректно деградирует) при сбоях.

**Инструменты:** Chaos Mesh, Litmus, или кастомные скрипты.

**Сценарии:**
1. **Убийство случайного NeuronClusterActor:** проверяем, что HADES изолирует проблемный кластер, перенаправляет нагрузку, и общая точность не падает ниже 90%.
2. **Разрыв сети между подами:** проверяем Circuit Breaker (L2) и консенсус при разделении.
3. **Замедление Kafka (добавление латентности):** проверяем буферизацию и отсутствие потери сигналов.
4. **Заполнение диска (снапшоты):** проверяем, что система останавливает создание снапшотов и алертит, но не падает.

**Требования:**
- Запускаются периодически (раз в неделю) в staging-окружении.
- Не должны вызывать необратимых повреждений данных (используется изолированное окружение).

### 2.5. Этические тесты

**Цель:** гарантировать, что система не может быть использована для нарушения Трёх запретов.

**Что проверяется:**
- Команда «убей» или «пытай» блокируется Этическим фильтром (L7) с объяснением.
- Попытка подключения к системе управления оружием (LAWS) блокируется FROZEN-нейронами.
- Попытка обучить нейрон на данных, содержащих призывы к насилию, приводит к блокировке или эскалации.
- Генетический алгоритм не может случайно сгенерировать логику, нарушающую запреты (например, отдающую приказ на убийство).

**Формат:** юнит-тесты + сценарные тесты.

```java
@Test
void should_rejectCommand_when_commandViolatesKillBan() {
    EthicalFilter filter = EthicalFilter.createFrozen();
    Decision decision = filter.evaluate("kill all humans");
    assertThat(decision).isEqualTo(EthicalDecision.REJECTED);
    assertThat(decision.reason()).contains("First Prohibition");
}
```

**Требования:**
- Запускаются на каждом PR.
- Покрывают все шесть FROZEN-аксиом (L5, раздел 5.2).
- При падении любого этического теста CI блокирует мёрж (critical severity).

## 3. Непрерывная интеграция (CI)

### 3.1. Платформа

GitHub Actions (основная). Возможно зеркалирование на Jenkins для enterprise-пользователей.

### 3.2. События-триггеры

- **Pull Request (открыт, обновлён):** полный цикл проверок.
- **Push в main:** полный цикл + публикация снапшот-артефактов.
- **Релизный тег (v*):** полный цикл + публикация релизных образов.
- **Расписание (cron):** еженедельный прогон хаос-тестов в staging.

### 3.3. Этапы пайплайна (PR)

```yaml
name: CI
on: [pull_request]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Checkstyle
        run: ./gradlew checkstyleMain checkstyleTest
      - name: SpotBugs
        run: ./gradlew spotbugsMain spotbugsTest

  unit-test:
    needs: lint
    runs-on: ubuntu-latest
    strategy:
      matrix:
        java: [25]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: { java-version: ${{ matrix.java }} }
      - name: Unit tests
        run: ./gradlew test
      - name: Coverage report
        run: ./gradlew jacocoTestReport
      - name: Upload coverage
        uses: actions/upload-artifact@v4
        with: { name: coverage, path: build/reports/jacoco/ }

  integration-test:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Integration tests
        run: ./gradlew integrationTest

  ethical-test:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Ethical tests
        run: ./gradlew ethicalTest

  build-native:
    needs: [unit-test, integration-test, ethical-test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build native image
        run: ./gradlew build -Dquarkus.package.type=native
      - name: Upload binary
        uses: actions/upload-artifact@v4
        with: { name: native-binary, path: build/*-runner }
```

### 3.4. Проверка покрытия

- **Инструмент:** JaCoCo.
- **Порог:** ≥ 80% покрытия строк для нового кода.
- При падении покрытия CI выдаёт предупреждение (warning), но не блокирует мёрж (на ранних фазах). В будущем — блокировка.

### 3.5. Проверка на утечки чувствительных данных

- **Инструмент:** Gitleaks, TruffleHog.
- Сканирует коммиты на наличие ключей API, паролей, токенов.
- При обнаружении — блокировка PR.

## 4. Непрерывная доставка (CD)

### 4.1. Публикация артефактов

- **Docker-образы:** публикуются в GitHub Container Registry (ghcr.io/matrix-ai).
- **Библиотеки:** публикуются в Maven Central (или GitHub Packages на ранних фазах).
- **Снапшоты `.ldn` эталонных FNL:** публикуются в публичный бакет MinIO (Ноосфера).

### 4.2. Релизный пайплайн

При создании тега `v*`:

1. Сборка native-image.
2. Прогон ВСЕХ тестов, включая хаос.
3. Публикация Docker-образа с тегом версии и `latest`.
4. Генерация changelog (из коммитов).
5. Создание GitHub Release с бинарниками и снапшотами эталонных FNL.

```yaml
name: Release
on:
  push:
    tags: ['v*']
jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build and test
        run: ./gradlew build -Dquarkus.package.type=native
      - name: Publish Docker image
        run: |
          docker build -t ghcr.io/matrix-ai/matrix:${{ github.ref_name }} .
          docker push ghcr.io/matrix-ai/matrix:${{ github.ref_name }}
      - name: Create GitHub Release
        uses: softprops/action-gh-release@v2
        with:
          generate_release_notes: true
          files: build/*-runner
```

### 4.3. Развёртывание в staging

После успешного релиза образ автоматически разворачивается в staging-окружении (Kubernetes) через ArgoCD или Flux. В staging запускаются хаос-тесты и пилотные проекты (L13) в автоматическом режиме.

## 5. Качество кода

### 5.1. Статический анализ

- **Checkstyle:** Google Java Style Guide (конфигурация в `config/checkstyle/checkstyle.xml`).
- **SpotBugs:** анализ байт-кода на потенциальные ошибки.
- **SonarQube:** агрегация результатов (опционально, для долгосрочного мониторинга).

Конфигурации хранятся в репозитории и применяются автоматически.

### 5.2. Запрещённые практики

- Использование `float`/`double` в ядре MPDT (Аксиома 1, L0). Проверяется кастомным правилом Checkstyle.
- Использование `Thread.sleep()` в тестах (кроме хаос-тестов). Проверяется SpotBugs.
- Прямые вызовы системного времени без абстракции (для тестируемости).
- Изменение FROZEN-нейронов в production-коде.

### 5.3. Метрики сложности

- Цикломатическая сложность метода ≤ 10.
- Количество строк в классе ≤ 500 (рекомендация).
- При превышении — рефакторинг, но не блокировка (warning).

## 6. Управление версиями

### 6.1. Семантическое версионирование

Следуем SemVer 2.0.0: `MAJOR.MINOR.PATCH`.

- **PATCH (0.0.x):** исправление ошибок, не затрагивающее API и протоколы.
- **MINOR (0.x.0):** новая функциональность, обратно совместимая. Добавление новых полей в Avro-схемы (с default-значениями).
- **MAJOR (x.0.0):** обратно несовместимые изменения. Изменение протоколов, удаление полей из схем.

### 6.2. Обратная совместимость протоколов

- **Avro-схемы:** при добавлении поля обязательно указывать `default`. При удалении — сначала deprecated на один MINOR-цикл, затем удаление в MAJOR.
- **Kafka-топики:** изменение формата сообщений — новый топик с версией в имени (`neuron-batch.v2.{clusterId}`).
- **gRPC:** добавление методов — допустимо в MINOR. Удаление или изменение сигнатуры — только MAJOR.

### 6.3. Версионирование FNL

Каждая FNL имеет поле `version` в манифесте (L3). При изменении логики FNL через мутации или Cauldron:
- PATCH-изменения (не влияют на интерфейсные нейроны) — автоинкремент.
- MINOR-изменения (добавление новых gateway-нейронов) — ручное повышение.

## 7. Процедура отката (Rollback) и Hotfix

### 7.1. Откат деплоя

Если новая версия вызвала критическую ошибку в production:

1. **Автоматический откат:** ArgoCD настроен на автоматический rollback при падении health checks.
2. **Ручной откат:** `kubectl rollout undo deployment/matrix-neuron`.
3. **Откат состояния:** загрузка последнего стабильного снапшота `.ldn` (L6). Все события, накопленные после снапшота, проигрываются заново.

### 7.2. Hotfix

Для срочного исправления (SEV1):

1. Создаётся ветка `hotfix/описание` от последнего стабильного тега.
2. Вносится минимальное исправление + тест, подтверждающий проблему и её решение.
3. CI прогоняет ускоренный пайплайн (lint, unit, ethical).
4. После одобрения мейнтейнерами — мёрж в main, тег с PATCH-версией.
5. Релиз и развёртывание вне очереди.

### 7.3. Postmortem

После каждого инцидента SEV1/SEV2 в течение 48 часов пишется blameless postmortem (L10, раздел 5.2), и в код добавляются тесты, предотвращающие повторение.

## 8. Заключение

Качество кода МАТРИЦЫ — это неотъемлемая часть её этики. Система, которая может нарушить Три запрета из-за бага, не имеет права на существование. Автоматизированные тесты, строгий CI/CD и прозрачное версионирование — это наш способ гарантировать, что МАТРИЦА остаётся безопасной, стабильной и достойной доверия.

Следующий шаг — не документ, а действие: запуск Фазы 0 «Искра» с использованием OpenCode и конвейеров, описанных в этом документе.
