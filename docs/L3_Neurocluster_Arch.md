# L3 — Архитектура нейронных кластеров и функциональных нейронных долей (FNL)

**Дата:** 2026-05-31
**Статус:** Утверждён
**Назначение:** Определить архитектуру нейронного кластера — основной структурной единицы исполнения, и функциональной нейронной доли — специализированной надстройки для решения когнитивных задач.

---

## 1. Введение

### 1.1. Назначение документа

Настоящий документ определяет архитектуру **Нейронного Кластера (Neuron Cluster)** и **Функциональной Нейронной Доли (Functional Neural Lobe, FNL)** в системе МАТРИЦА. Нейронный кластер — это минимальная единица масштабирования и управления миллионами MPDT-нейронов. FNL — специализированная логическая группа кластеров, обученная под конкретную задачу и способная динамически загружаться и выгружаться.

Документ опирается на формальное определение MPDT-нейрона (L1) и протоколы взаимодействия (L2).

### 1.2. Ключевые принципы

- **Масштабируемость через кластеризацию:** группировка миллионов нейронов в один актор минимизирует накладные расходы.
- **Иерархическая композиция:** кластеры объединяются в FNL; FNL могут вкладываться в более крупные структуры.
- **Динамическая загрузка:** FNL загружаются и выгружаются по требованию, экономя ресурсы.
- **Пакетный инференс и локальная эволюция:** все вычисления внутри кластера оптимизированы для массовой обработки сигналов и фоновой оптимизации.

## 2. Архитектура нейронного кластера

### 2.1. Определение

**Нейронный кластер** — это программный компонент, реализованный как Pekko Cluster Actor, который владеет пулом MPDT-нейронов и выполняет их инференс, обучение и взаимодействие с внешним миром.

Кластер характеризуется:
- **ClusterId** — глобально уникальный идентификатор (UUID).
- **Роль (Role):** `"sensor"`, `"wm"`, `"ltm"`, `"motor"`, `"mediator"` — в соответствии с архитектурными ролями.
- **Поколение (Generation):** увеличивается при каждом значимом изменении топологии или состава нейронов.
- **Вектор возможностей (Capability Vector):** набор ключ-значение, описывающий, какие FNL и навыки доступны данному кластеру.

### 2.2. Внутренняя структура NeuronClusterActor

Каждый инстанс `NeuronClusterActor` хранит:

1. **Пул нейронов (Neuron Pool):**
   - `activeNeurons: Map<NeuronId, NeuronInstance>` — для состояний `STABLE` и `FROZEN`; быстрый доступ по ID.
   - `pendingMutations: List<MutationTrial>` — для нейронов в состоянии `MUTATING` или `LEARNING`.
2. **Локальный кэш топологии (Topology Cache):** `Map<NeuronId, List<NeuronId>>` — список входов для каждого нейрона; позволяет быстро вычислять пакеты входных сигналов.
3. **Очередь входящих сигналов (Input Queue):** кольцевой буфер для приёма пакетов из Kafka перед обработкой.
4. **Исходящий буфер (Output Buffer):** накопитель сигналов для отправки в другие кластеры (реализует Outbox Pattern).
5. **Журнал событий кластера (Cluster Event Log):** локальная Kafka-тема для записи всех мутаций и структурных изменений.
6. **Статистика (Metrics):** скользящие окна точности, задержек, энергопотребления, числа ошибок.

### 2.3. Жизненный цикл нейрона внутри кластера

Нейрон в кластере проходит через следующие фазы (соответствуют состояниям из L1):

| Состояние | Описание | Где хранится |
|-----------|----------|--------------|
| `LEARNING` | Мягкое дерево, настройка на данных | `pendingMutations` |
| `MUTATING` | Эволюционные испытания мутаций | `pendingMutations` |
| `STABLE` | Готовая булева функция, используется в инференсе | `activeNeurons` |
| `FROZEN` | Неизменяемый нейрон (аксиомы безопасности) | `activeNeurons` с флагом `immutable=true` |

При переходе из `LEARNING`/`MUTATING` в `STABLE` нейрон детерминируется (компиляция в таблицу истинности) и переносится в `activeNeurons`.

## 3. Пакетный инференс внутри кластера

### 3.1. Основной цикл

Каждый `NeuronClusterActor` имеет рабочий цикл, выполняемый на виртуальных потоках Java:

1. **Приём пакетов:** из Input Queue извлекаются все пакеты, пришедшие за последний интервал `T_cycle` (настраиваемый, по умолчанию 1 мс).
2. **Распаковка и маршрутизация:** сигналы из пакетов направляются к целевым нейронам; обновляется кэш входных значений `currentInputs: Map<NeuronId, BitSet>`.
3. **Вычисление:** все активные нейроны, чьи входы изменились, пересчитывают своё состояние. Для эффективности используется пакетный вызов `evaluateBatch(neurons, inputs)`.
4. **Генерация исходящих сигналов:** нейроны, чей выход изменился, помещают свои сигналы в Output Buffer, сгруппированные по целевым кластерам.
5. **Отправка:** исходящий буфер сбрасывается в Kafka согласно Outbox Pattern (атомарно с обновлением состояния нейронов).

### 3.2. Пакетный вызов evaluateBatch

Для ускорения инференса используется подход, аналогичный SIMD (Java Vector API). Таблица истинности каждого нейрона хранится как `long[]` (биты упакованы). Входной индекс вычисляется путём сдвига и побитового OR. Затем выполняется прямое обращение к массиву.

```
// Псевдокод
void evaluateBatch(Neuron[] neurons, InputCache inputs) {
    for (Neuron n : neurons) {
        int index = computeIndex(n, inputs);
        boolean out = (n.truthTable[index >>> 6] >>> (index & 63) & 1) == 1;
        if (out != n.lastOutput) {
            n.lastOutput = out;
            emitSignal(n.id, out);
        }
    }
}
```

## 4. Функциональная нейронная доля (FNL)

### 4.1. Определение

**FNL** — это логическая группа из одного или нескольких нейронных кластеров, специализированная для выполнения определённой когнитивной функции (зрение, синтез речи, планирование, кратковременная память). FNL имеет:

- Уникальный `LobeId`.
- Манифест (`LobeManifest`), описывающий её возможности, зависимости, требования к ресурсам и протокол взаимодействия.
- Набор правил маршрутизации для подключения к другим FNL.
- Встроенный **LobeMediator** — облегчённый актор, координирующий работу нейронов внутри FNL.

### 4.2. Just-In-Time Cognitive Loading

FNL могут динамически загружаться и выгружаться с диска или из сети под управлением вышестоящего Медиатора. Это аналогично тому, как мозг активирует нужные зоны для текущей задачи, а не держит все нейроны постоянно активными.

**Процесс загрузки FNL:**

1. Медиатор получает запрос или внутренний драйвер, требующий определённой когнитивной функции.
2. Медиатор ищет в реестре FNL, способную выполнить функцию. Если она не загружена, отправляет команду `LoadFNL` на целевой кластер.
3. Кластер загружает снапшот FNL из распределённого хранилища (S3/MinIO) или от другого узла. Снапшот содержит состояние всех нейронов доли и её топологию.
4. После загрузки FNL интегрируется в локальный инференс-цикл кластера.
5. По завершении задачи или при нехватке ресурсов FNL может быть выгружена, а её состояние синхронизировано с хранилищем.

**Преимущества:**
- Экономия памяти и энергии: активны только необходимые доли.
- Безграничная расширяемость: новые способности добавляются как новые FNL без перестройки системы.
- Специализация: FNL может быть обучена под конкретного пользователя или окружение.

### 4.3. Внутренняя структура FNL

FNL состоит из:

- Одного или нескольких нейронных кластеров (или их частей), связанных локальными связями.
- **Интерфейсных нейронов (Gateway Neurons):** небольшая группа нейронов, через которые FNL принимает сигналы извне и отправляет результаты. Они имеют глобальные входы/выходы, в то время как остальные нейроны FNL связаны только локально.
- **Локального Медиатора (LobeMediator):** облегчённый актор, координирующий работу нейронов внутри FNL, следящий за её метриками и взаимодействующий с вышестоящим ClusterMediator.

### 4.4. LobeMediator (Микро-Медиатор FNL)

LobeMediator — это облегчённая версия Медиатора с ограниченным набором драйверов (Curiosity и Entropy) и правом принимать локальные решения о мутациях в пределах своей FNL.

**Ограничения LobeMediator:**
- Не может изменять топологию связей с другими FNL.
- Не может инициировать проактивный диалог с пользователем.
- Не может изменять свой вес — он фиксирован.
- Все мутации, затрагивающие более 10 нейронов, должны быть одобрены ClusterMediator.

## 5. Взаимодействие с Медиатором

### 5.1. Команды Медиатора

Медиатор посылает кластеру управляющие команды через выделенный топик `cluster.control.{clusterId}`. Типы команд:

| Команда | Описание |
|---------|----------|
| `LoadFNL(lobeId, manifest)` | Загрузить FNL из хранилища |
| `UnloadFNL(lobeId)` | Выгрузить FNL, сохранив состояние |
| `StartMutation(neuronId, strategy)` | Запустить мутацию нейрона |
| `SetDriverPriority(driver, value)` | Изменить вес драйвера |
| `ReportStatus()` | Запросить метрики и состояние |
| `Freeze(neuronId)` | Заморозить нейрон (перевести в FROZEN) |
| `Shutdown()` | Плановое выключение с сохранением состояния |

### 5.2. Отчётность кластера

Кластер периодически отправляет Медиатору свои метрики через топик `cluster.metrics`:
- Загрузка CPU и памяти.
- Пропускная способность инференса (сигналов/сек).
- Точность и энтропия.
- Состояние очередей и буферов.
- Активные FNL и их потребление ресурсов.

## 6. Масштабирование и балансировка

### 6.1. Шардирование нейронов

Для масштабирования до триллионов нейронов используется **шардирование (Pekko Cluster Sharding)**. Нейроны распределяются по шардам на основе хеша от их `NeuronId`. Шард — это группа нейронов, обслуживаемая одним инстансом `NeuronClusterActor`.

**Координатор шардов (Shard Coordinator):** singleton-актор, отслеживающий, на каком узле находится каждый шард, и перенаправляющий запросы при переезде шарда.

### 6.2. Балансировка нагрузки

При перегрузке одного узла или неравномерном распределении FNL, Медиатор может инициировать миграцию шардов. Миграция происходит без остановки сервиса:

1. Текущий узел сохраняет снапшот состояния шарда.
2. Снапшот передаётся на целевой узел.
3. Восстанавливается состояние, и шард регистрируется на новом месте.
4. Старый узел удаляет локальные данные.

### 6.3. Горизонтальное масштабирование

Новые узлы могут присоединяться к кластеру в любой момент. Pekko Cluster обнаруживает их, и шарды автоматически перераспределяются для равномерной загрузки.

## 7. Derangement Detection и Cauldron Protocol

### 7.1. Derangement Detection (Обнаружение одичания)

Если FNL или кластер демонстрирует устойчивое отклонение от ожидаемого поведения (рост энтропии выше критического порога, игнорирование команд Медиатора, генерация противоречивых сигналов), система регистрирует состояние "Derangement".

**Действия при обнаружении:**
1. Кластер помечается как `QUARANTINED`.
2. Все сигналы от него проходят двойную проверку.
3. InstanceMediator получает уведомление и может инициировать процедуру HADES (контролируемый перезапуск, описанный в L2).
4. Остальная система временно маршрутизирует запросы в обход проблемного компонента.

### 7.2. Cauldron Protocol (Рождение новых FNL)

Когда Медиатор обнаруживает, что существующие FNL не справляются с новым классом задач, он может инициировать Cauldron:

1. **Выделение песочницы:** резервируется изолированный кластер, не имеющий доступа к Этическому фильтру и внешним интерфейсам.
2. **Генерация семени:** создаётся популяция из `P_seed` (по умолчанию 1000) случайных MPDT-нейронов.
3. **Ускоренная эволюция:** запускается генетический алгоритм с агрессивными параметрами. Фитнес-функция — точность на целевом наборе данных.
4. **Закалка:** лучшие нейроны переводятся в `STABLE` и объединяются в минимальную сеть.
5. **Аудит:** новая FNL проверяется Этическим фильтром.
6. **Регистрация:** FNL получает `LobeId`, манифест и добавляется в реестр.

## 8. Обработка отказов и живучесть

### 8.1. Отказ NeuronClusterActor

При падении узла или актора:
- Pekko Persistence восстанавливает состояние актора из журнала событий (Event Sourcing) и последнего снапшота.
- Пропущенные за время простоя сигналы могут быть восстановлены из Kafka.
- Нейроны в состояниях `LEARNING` или `MUTATING` могут потерять несохранённые испытания, что допустимо, так как они не влияют на стабильную работу.

### 8.2. Перегрузка

Если очередь входящих сигналов кластера растёт быстрее, чем обрабатывается, активируется механизм обратного давления (backpressure): кластер посылает сигнал отправителям о необходимости снизить частоту. Если это не помогает, Медиатор может загрузить дополнительные FNL для разгрузки.

## 10. Заключение

Документ L3 описывает, как из элементарных MPDT-нейронов строятся масштабируемые, отказоустойчивые и динамически адаптируемые структуры — нейронные кластеры и функциональные нейронные доли. Пакетный инференс внутри кластера обеспечивает высокую пропускную способность, а Just-In-Time Cognitive Loading и Cauldron Protocol позволяют системе наращивать когнитивные способности без перезапуска. Механизмы Derangement Detection и миграции шардов гарантируют живучесть даже при сбоях отдельных компонентов.
