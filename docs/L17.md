## 2. Первый контрибьюторский путь за 30 минут

### 2.1. Шаг 1: Установка и запуск (10 минут)

```bash
# Клонируйте репозиторий
git clone https://github.com/matrix-ai/matrix.git
cd matrix

# Запустите сборку (убедитесь, что JDK 25 и Gradle 8+ установлены)
./gradlew build

# Запустите тесты
./gradlew test
```

Если всё зелёное — вы готовы. Если что-то не работает, загляните в раздел «Помощь» ниже.

### 2.2. Шаг 2: Найдите задачу (5 минут)

1. Перейдите на вкладку Issues в GitHub-репозитории.
2. Отфильтруйте по метке `good first issue` — это задачи, специально отобранные для новичков.
3. Выберите то, что вам интересно, и оставьте комментарий: «Берусь за эту задачу». Если задача никому не назначена, она ваша.
4. Если подходящей задачи нет, спросите в чате (Matrix/Discord) или создайте новый Issue с предложением.

**Примеры задач для первого вклада:**
- Исправить опечатку в документации.
- Написать юнит-тест для неохваченного метода.
- Улучшить сообщение об ошибке.
- Перевести часть документации на другой язык.

### 2.3. Шаг 3: Внесите изменения (10 минут)

1. Создайте форк репозитория и склонируйте его.
2. Создайте ветку: `git checkout -b feature/короткое-описание`.
3. Внесите изменения, следуя правилам оформления кода (раздел 5).
4. Закоммитьте: `git commit -m "feat: краткое описание изменения"`.
5. Запушьте: `git push origin ваша-ветка`.

### 2.4. Шаг 4: Откройте Pull Request (5 минут)

1. Перейдите в оригинальный репозиторий на GitHub, нажмите «New Pull Request».
2. Выберите вашу ветку, заполните описание:
   - Что сделано.
   - Ссылку на Issue (если есть).
   - Любые вопросы или замечания для ревьюера.
3. Нажмите «Create Pull Request».

### 2.5. Что дальше

Ваш PR будет автоматически проверен CI (тесты, стиль кода). Затем один из мейнтейнеров проведёт ревью. Не пугайтесь замечаний — это нормальная часть процесса. Вносите правки, и скоро ваш код станет частью МАТРИЦЫ.

- `good first issue` — лучшие задачи для первого вклада.

- `bug` — исправление ошибок.

### 3.2. Обсуждения (GitHub Discussions)

- Раздел `Ideas` — предложите новую идею, обсудите с сообществом.

### 3.3. Roadmap

В документе L8 (раздел 3) описаны фазы разработки. Вы можете взять задачу из ближайшей фазы, особенно если она не назначена другим.

## 4. Правила оформления кода

### 4.1. Стиль кода

- **Java:** следуем Google Java Style Guide. Конфигурация Checkstyle включена в сборку.
- **Отступы:** 4 пробела (не табуляция).
- **Имена:** классы — CamelCase, методы и переменные — camelCase, константы — UPPER_SNAKE_CASE.
- **Максимальная длина строки:** 120 символов.

### 4.2. Документирование

- Все публичные классы и методы должны иметь Javadoc с описанием назначения, параметров и возвращаемого значения.
- Ссылайтесь на соответствующие разделы спецификаций L1–L8, если это уместно.

```java
/**
 * Вычисляет выход MPDT-нейрона для заданного входного вектора.
 *
 * @param input входной битовый вектор длины k
 * @return булево значение выхода
 * @see L1, раздел 3.1
 */
public boolean evaluate(BitSet input) {
    // ...
}
```

### 4.3. Тесты

- Новый функционал должен сопровождаться тестами.
- Используйте JUnit 5 и AssertJ.
- Именуйте тесты по шаблону: `should_expectedBehavior_when_condition`.
- Стремитесь к покрытию ≥ 80% для нового кода.

### 4.4. Коммиты

- Используйте осмысленные сообщения на русском или английском (единообразно в рамках одного PR).
- Рекомендуемый формат: `тип: краткое описание`, где тип: `feat`, `fix`, `docs`, `test`, `refactor`, `chore`.
- Один коммит — одно логическое изменение. Если изменений много, разбейте на несколько коммитов.

## 5. Процесс ревью

### 5.1. Кто ревьюит

Ревью проводят мейнтейнеры модуля, в который вы вносите изменения. Для мелких правок достаточно одного ревьюера, для значительных изменений — два или более.

### 5.2. Сроки

Мы стараемся провести первичное ревью в течение 48 часов (рабочие дни). Если задержка больше, вежливо напомните о себе в комментариях к PR.

### 5.3. Как реагировать на замечания

- Замечания — это не критика вас лично, а способ улучшить код.
- Если вы не согласны с замечанием, обсудите его аргументированно.
- Вносите правки в той же ветке новыми коммитами. После обновления ветки PR обновится автоматически.

### 5.4. Одобрение и мёрж

Когда все замечания сняты и тесты проходят, ревьюер одобряет PR и выполняет мёрж. Ваш код вливается в основную ветку. Поздравляем — вы контрибьютор МАТРИЦЫ!

## 6. Как предложить крупную идею (RFC)

Если ваша идея затрагивает архитектуру, протоколы или спецификации (L0–L8), используйте процесс RFC:

1. Создайте обсуждение в GitHub Discussions (категория `Ideas`) или Issue с пометкой `rfc`.
2. Опишите:
   - Проблему, которую вы решаете.
   - Предлагаемое решение.
   - Влияние на существующую архитектуру.
   - Альтернативы, которые вы рассматривали.
3. Дайте сообществу минимум 2 недели на обсуждение (для архитектурных вопросов — до 1 месяца, согласно L11).
4. Если достигнут консенсус, приступайте к реализации.
